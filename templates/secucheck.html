5. 보안 설정 오류
(Security Misconfiguration)
정의
기본적으로 탑재 되어야 하는 보안 구성이 제대로 적용되어 있지 않은 경우. 애플리케이션에서 사용되는 소프트웨어가 최신 상태가 아닐 경우에 발생할 수 있다.
종류
  • 디렉터리 인덱싱
  • 불필요한 지원
  • 취약한 파일 존재
  • 히든필드 조작
  • 알려진 취약점이 있는 구성요소 사용
공격기법
1. 디렉터리 인덱싱
일반적으로 웹 서버에는 URL 입력 시 Default.html, index.html, index.html 등 기본적으로 호출할 파일을 지정해 놓게 되는데, 웹 루트 디렉토리에 해당 파일이 없을 경우, 디렉토리 내의 파일 리스트를 웹브라우저에 표시하게 된다. 디렉토리 인덱싱 취약점은 웹 서버 내 특정 디렉터리 파일 리스트를 노출하고 공격자는 이를 이용하여 응용 시스템의 구조를 파악하고, 주요 소스 파일 또는 설정 파일을 다운로드 하는 등 정보를 획득할 수 있다.
2. 불필요한 Method 지원
Method는 목적에 따라 GET, POST, HEAD, PUT, DELETE, OPTIONS 등으로 분류되며 사전에 정의된 행위를 수행하게 된다. 이 중 PUT, DELETE Method의 경우 임의로 서버 내 파일의 생성 및 삭제가 가능하기 때문에 공격자에 의한 조작의 위험이 있다.




3. 취약한 파일 존재
웹 루트 하위에 내부 문서나 백업파일, 로그파일, 압축파일과 같은 파일이 존재할 경우 파일명을 유추하여 파일명을 알아내고, 직접 요청하여 해킹에 필요한 서비스 정보를 획득할 수 있는 취약점.


4. 히든필드 조작
  • Hidden Field는 사용자에게는 보여 지지 않지만 서버와 클라이언트간에 데이터를 교환에 사용되는 일종의 이름/값 쌍을 가지는 데이터 필드다.

  • Hidden Field의 인자를 조작하거나 숨겨진 기능을 찾아 애플리케이션의 비정상적인 동작을 유도한다.

  • 승인되지 않은 페이지에 접근 가능하거나 애플리케이션 오류를 발생시켜 정보 획득 및 추가 공격이 가능하다.

5. 알려진 취약점이 있는 구성요소 사용
  • 웹 애플리케이션에서 취약점이 알려진 라이브러리, 프레임워크, 모듈 등을 사용하여 발생할 수 있는 취약점



점검내용
1. 디렉터리 인덱싱
  • 직접 점검 방법
    • - 가) 점검 대상 웹 사이트의 하위 디렉터리 정보를 사전에 모두 확인 - 나) 웹 루트(Root)의 모든 하위 디렉터리에 대해서 웹 브라우저에 해당 주소를 입력하여 디렉토리 취약점 존재여부를 점검
  • 구글을 통한 점검
    • - 도메인 설정 란에 해당 사이트 주소를 입력하여 디렉터리 목록이 저장된 페이지를 검색, 해당 사이트의 디렉터리가 노출되었는지 확인
2. 불필요한 Method 지원
  • 텔넷에 접속하여 불필요한 Method가 활성화 되어 있는지 확인
3. 취약한 파일 존재
  • 웹서버의 가상 디렉터리로 이동하여 다음에 제시되는 확장자의 파일을 찾아 불필요한 정보가 포함 되었는지 여부를 판단
  • PHP 언어로 개발한 웹서버의 경우 아래와 같은 정보를 출력하는 웹페이지 (phpinfo.php)가 존재하는지 점검
4. 히든필드 조작
  • 테스트 계정, default 계정 등 취약한 계정 사용 여부 확인
5. 알려진 취약점이 있는 구성요소 사용
  • 웹 애플리케이션 및 상용 소프트웨어의 버전 확인
대응방법
1. 디렉터리 인덱싱
  • Windows의 IIS 조치 : '인터넷 정보서비스(IIS)'를 이용하는 웹서버이며 작업을 통해 디렉터리 인덱싱 취약점 차단 가능
  • Unix, Linux의 Apache 조치 : 유닉스 및 리눅스 운영체제로서 아파치(Apache)를 이용하는 웹서버이며 아래 작업을 통해 디렉터리 인덱싱 취약점을 차단 가능
  • 파일 위치를 모를 경우 : 유닉스 및 리눅스 운영체제로서 톰캣(Tomcat)을 이용하는 웹서버이며 아래 작업을 통해 디렉터리 인덱싱 취약점을 차단 가능
2. 불필요한 Method 지원
  • 웹 서버 내에서의 조치
    • - 가) 홈페이지 운영에 불필요한 Method(PUT, DELETE, OPTIONS 등) 비활성화 - 나) 웹 서버 홈 디렉터리 아래 conf 디렉터리의 web.xml파일에서 불필요한 Method 비활성화 (삭제 또는 주석)를 위한 보안설정
3. 취약한 파일 존재
  • 가) 웹서버는 개발과 운영 환경을 분리하여 운영 환경에서 소스 코드 수정 또는 테스트 목적의 임시 파일을 생성하지 않도록 한다
  • 나) 웹 서버의 디렉터리에 존재하는 기본 설치 파일, 임시 및 백업 파일을 조사하여 웹 사용자가 접근하지 못하도록 조치
  • 다) 정기적으로 불필요 파일을 검색하여 제거한다


4. 히든필드 조작
  • 홈페이지 개발 보안 조치
    • - 가) 중요 정보(개인정보, 패스워드 등)는 클라이언트에서 확인 가능한 소스코드 내의 Hidden Field에서의 저장을 지양하고, 저장이 불가피할 경우 해당 내용은 암호화 처리
    - 나) 중요한 정보 및 숨겨진 기능 처리는 서버에서 이루어질 수 있도록 설계
    - 다) Hidden Field 데이터에 대한 입력 값 검증 로직을 구현하여 무결성 체크
5. 알려진 취약점이 있는 구성요소 사용
  • 웹 서버 내에서의 조치
    • - 가) 웹 서버에는 공개용 웹 게시판 사용 지양
    - 나) 사용 시 보안 패치 또는 최신버전의 제품으로 설치하며, 정 기적으로 게시판 배포 사이트에 방문하여 보안 취약점 정보를 확인
    - 다) 서버 정보 노출 방지
